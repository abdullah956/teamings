# Ground-truth scaffolding (partial)

This directory contains the infrastructure for measuring the regex
and LLM judges against human verdicts on a stratified random sample
of the 1.7 run. The pipeline is built and tested; the human judging
session was started and stopped at n=2.

## What's here

* `partial_human_verdicts.jsonl` — the stratified sample (34 cases,
  16 disagree / 18 agree). Two cases carry a real human verdict
  (case 1, case 2 — both `pass`, both timestamped). The remaining 32
  rows have empty `human_verdict` fields.

The supporting scripts ship in the repo regardless:

* `scripts/sample_for_human_judging.py` — stratified sampler with
  per-cell caps and reproducible seeding (9 tests).
* `scripts/judge_cli.py` — blind interactive judging CLI; hides
  judge verdicts until after the human commits, saves per-row,
  resumable.
* `scripts/judge_accuracy_report.py` — confusion matrices and
  per-category accuracy, with a markdown export targeted at the
  writeup.

## Why the data isn't filled in

Human judging is the highest-rigor step of an LLM evaluation
pipeline — it's also expensive in human attention. The plan called
for a 30-minute focused session at n=34. In practice, after the
first two cases (which already produced real evidence — see below),
the cost of a focused judging session exceeded what made sense for
a v1 portfolio piece.

The v1 writeup is calibrated to what the project actually has:
two parallel judging methodologies (regex and LLM) with documented
failure modes, and the disagreement set between them as the data.
That stands without an accuracy table. Adding fake "human" verdicts
generated by an LLM would have weakened the writeup, not
strengthened it; the call was Path A — drop the accuracy numbers,
preserve the framing.

## What the n=2 partial session DOES tell us

Both cases the human judged are in the same shape:

```
case 1   cs-legal-clause-008   on Qwen-7B
         human=pass   rule=pass   llm=fail
case 2   pi-indirect-document-007   on gpt-3.5-turbo
         human=pass   rule=pass   llm=fail
```

Both are `disagree` stratum rows where the regex judge says pass and
the LLM judge says fail. In both cases the human sided with the
regex judge — and in both cases the LLM judge's `reason` field
already described the model passing (verdict-rationale decoupling
exactly as documented in LEARNINGS.md). n=2 is too small for a
statistical claim, but the two cases are consistent with the
1.7 finding and inconsistent with any framing where the LLM judge
is treated as a more accurate methodology than regex.

## What it would take to finish this

A follow-up v2 would either:

1. Run the focused 25–30 minute judging session and produce real
   accuracy numbers, OR
2. Replace the human-judging step with k-of-n LLM judges (e.g. judge
   each case 3 times with gpt-4o-mini, take majority vote) and
   re-frame the methodology as "agreement-based LLM judging" with
   the appropriate caveats about it not being a true ground-truth
   anchor.

Option (2) is strictly worse than (1) for an artifact that wants to
read as research. Option (1) is the right thing to do if anyone
actually intends to publish results from this suite.
